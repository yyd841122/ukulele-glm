# MONETIZATION · 尤克里里 AI 学园 — 商业化与会员体系设计

> **文档版本**：v1.0  ｜  **创建**：2026-06-26
> **配套**：`PRD.md` §8、`docs/TDD.md` §4 + 商业化架构章节、`ADR-002`
> **定位**：本文件是商业化的**唯一事实来源**。会员档位、权益矩阵、支付渠道、订单流程、退款规则全在这里定义，代码与原型据此实现。
> **核心策略**：**免费试用期 + 永久会员**，**安卓先行**，商店内购 + 微信/支付宝直连双渠道。

## 0. 商业化总策略

| 维度 | 决策 |
|------|------|
| **主力会员模式** | **永久会员**（一次付费，永久解锁尤克里里全部内容） |
| **转化抓手** | **免费试用期**（新用户注册赠 X 天全权益体验，到期降级） |
| **辅助变现** | 单内容付费（单曲/单课，后期） |
| **首发平台** | **Android 优先**；iOS 后期再决定（苹果抽成与审核成本高） |
| **支付渠道** | 安卓：**应用商店内购 + 微信/支付宝直连**（双轨）；iOS：苹果 IAP（后期） |
| **定价基调** | 显著低于对标产品（AI 音乐学园永久会员 ¥199–359），**建议永久会员 ¥99–149 测试** |

**为什么不主推订阅？** 尤克里里是「学会就毕业」的工具型品类，用户学习周期有限（通常 3–6 个月），订阅续费率会偏低；永久会员一次性转化、用户决策成本低，更契合品类特性。订阅作为后期可选项保留接口。

## 1. 会员档位体系

### 1.1 档位定义

| 档位（Tier） | 标识 | 获取方式 | 有效期 | 定价（建议测试区间） |
|-------------|------|---------|--------|-------------------|
| **免费版** | `FREE` | 注册即得 | 永久（受限） | ¥0 |
| **试用期** | `TRIAL` | 新用户注册赠送 | 注册后 **7 天** | ¥0（一次性） |
| **永久会员** | `LIFETIME` | 付费购买 | **永久** | **¥99–149**（A/B 测试） |
| _（远期）全品类 VIP_ | `VIP_ALL` | 付费购买 | 永久/订阅 | 待定（Phase 4 多乐器时启用） |

> 档位以**枚举 `Tier`** 固化在代码与数据中，新增档位只在枚举末尾追加，不删改已有值。

### 1.2 权益优先级（就高原则）
同一用户可能同时持有多个权益来源（如试用期 + 永久会员），生效权益取**优先级最高者**：
```
LIFETIME(永久) > VIP_ALL > TRIAL(试用) > FREE(免费)
```
后端权益校验接口返回「当前生效档位」= 用户所有有效权益中的最高档。

> 这张表是**功能权限控制的契约**，定义"谁能用什么"。所有功能页接入付费墙前必须先在此登记。

图例：✅ 可用 ｜ 🔒 会员专享 ｜ 💎 单独购买 ｜ ⏳ 试用期内全开

| 功能模块 | 功能项 | FREE 免费 | TRIAL 试用 | LIFETIME 永久 |
|---------|--------|:---------:|:---------:|:-------------:|
| **工具·调音器** | 基础调音 | ✅ | ✅ | ✅ |
| | High-G/Low-G 切换 | ✅ | ✅ | ✅ |
| **工具·节拍器** | 基础节拍 | ✅ | ✅ | ✅ |
| | 节拍细分/自定义节奏型 | 🔒 | ⏳ | ✅ |
| **工具·和弦库** | 基础和弦（C/Am/F/G/Em/Dm） | ✅ | ✅ | ✅ |
| | 进阶和弦全库 | 🔒 | ⏳ | ✅ |
| **曲谱库** | 入门曲目（⭐⭐ 及以下） | ✅ | ✅ | ✅ |
| | 进阶/指弹曲目（⭐⭐⭐+） | 🔒 | ⏳ | ✅ |
| | 高阶曲目（可改为💎单买） | 🔒 | 🔒 | ✅ / 💎 |
| **曲谱播放器** | 滚动播放/变速变调 | ✅ | ✅ | ✅ |
| | **跟弹 AI 评分** | 每日 3 次 | ⏳ 无限 | ✅ 无限 |
| **互动课程** | 必修课前 5 节 | ✅ | ✅ | ✅ |
| | 必修课全部 / 选修 / Pro 课 | 🔒 | ⏳ | ✅ |
| **游戏化** | 打卡/基础成就 | ✅ | ✅ | ✅ |
| | 排行榜/好友 PK | 🔒 | ⏳ | ✅ |
| **数据** | 本地数据看板 | ✅ | ✅ | ✅ |
| | 云端同步/多设备 | 🔒 | ⏳ | ✅ |

**设计原则**：
- **免费层足够有用**：调音、节拍、基础和弦、入门曲谱、每日 3 次评分都免费，保证留存与口碑（学琴刚需工具不卡）；
- **会员价值清晰**：核心变现点在「全部课程 + 全部曲谱 + 无限评分」；
- **跟弹评分限额**：免费用户每天 3 次，既让用户体验到核心价值，又制造升级动机。

## 3. 试用期（Trial）机制

| 项 | 规则 |
|----|------|
| 触发 | 用户**注册成功**自动开通 |
| 时长 | **7 天**（可配置） |
| 权益 | 试用期内享 LIFETIME 全部权益 |
| 到期处理 | 自动降级为 FREE，**不自动扣费**（无订阅，无支付信息，零合规风险） |
| 到期前提醒 | 第 5/6/7 天 App 内 + 推送提示「试用即将结束」 |
| 唯一性 | 一个设备/一个手机号仅享一次试用（防薅羊毛，见 §7 设备指纹） |
| 中断 | 试用期内购买永久会员 → 立即生效为 LIFETIME，试用期权益被覆盖 |

## 4. 支付渠道与订单

### 4.1 渠道矩阵（安卓优先）

| 渠道 | 平台 | 抽成 | 状态 | 备注 |
|------|------|------|------|------|
| **应用商店内购**（华为/小米/OPPO/vivo/应用宝） | Android | ~15–30% | ✅ 首发 | 走各商店 IAP SDK，订单由商店管理 |
| **微信支付** | Android（App 内 H5/JSAPI） | ~0.6% | ✅ 首发 | 直连，利润高，需自建订单系统 |
| **支付宝** | Android（App 内 H5） | ~0.6% | ✅ 首发 | 直连，同上 |
| 苹果 IAP | iOS | 30% | ⏳ 后期 | iOS 上架强制，暂缓 |

### 4.2 双渠道并存策略
安卓同时提供「商店内购」和「微信/支付宝直连」两个入口：
- **商店内购**：满足只习惯商店支付的用户，且利于应用商店推荐位（商店偏好内购 App）；
- **直连支付**：利润更高，且可作为商店抽成的补充/促销通道（如限时低价走直连）。
- 两渠道购买后**权益统一**：无论从哪买，后端校验订单成功即开通 LIFETIME 权益（见 §5 权益开通流程）。

### 4.3 订单状态机

```
            创建订单                支付成功回调
 CREATED ─────────► PENDING ─────────► PAID ──────► FULFILLED（权益已开通）
    │                  │                                 │
    │                  │ 支付失败/超时                     │ 用户申请退款
    │                  ▼                                 ▼
    └──────────────► FAILED/CANCELLED              REFUNDING ──► REFUNDED（权益回收）
```

| 状态 | 含义 | 是否触发权益 |
|------|------|------------|
| `CREATED` | 订单已建，未发起支付 | ❌ |
| `PENDING` | 已唤起支付，等待结果 | ❌ |
| `PAID` | 渠道回调确认收到钱 | 触发开通流程 |
| `FULFILLED` | 权益已写入用户账户 | ✅ |
| `FAILED` | 支付失败 | ❌ |
| `CANCELLED` | 用户取消 | ❌ |
| `REFUNDING` | 退款审核中 | 权益待定 |
| `REFUNDED` | 已退款 | ❌（回收权益） |

### 4.4 防掉单（关键）
支付渠道回调可能丢失，**必须做服务端主动对账**：
- 每笔订单写入 DB 后，启动**定时轮询**（如每 2 分钟）向渠道查询未 `FULFILLED` 订单状态；
- App 端支付完成回调后，主动调后端「校验订单」接口；
- 提供用户「恢复购买」入口（点击后端比对渠道订单 → 补开通）。

## 5. 权益开通与校验流程

### 5.1 开通流程（购买成功后）
```
渠道支付成功回调 ──► 后端校验订单签名/真实性 ──► 订单置 PAID
      │
      ▼
写入 Entitlement 表（user_id, tier=LIFETIME, source, order_id, granted_at）
      │
      ▼
订单置 FULFILLED ──► 推送"开通成功"通知 ──► App 刷新本地权益缓存
```

### 5.2 权益校验流程（每次访问受限功能）
```
用户访问功能 X
   │
   ▼
App 本地权益缓存（带过期时间，如 1 小时）
   ├─ 命中且为 LIFETIME ──► 放行
   ├─ 命中且为 FREE/TRIAL ──► 按 §2 矩阵判断：限额？放行 : 弹付费墙
   └─ 未命中/过期 ──► 调后端 /entitlements/check ──► 更新缓存 ──► 再判断
```

**校验原则**：
- **本地缓存优先**：减少网络依赖，离线也能用已开通权益；
- **敏感操作（如退款后）服务端强校验**：权益回收不能只靠客户端；
- **跟弹评分等限额功能**：限额计数在后端（按用户+天），客户端展示余额。

## 6. 退款规则

| 场景 | 规则 |
|------|------|
| **商店内购退款** | 遵循各商店退款政策；商店退款事件通过通知/对账同步后端 → 回收 LIFETIME 权益 |
| **微信/支付宝直连退款** | 7 天内未大量使用（如练习时长 < 阈值）支持全额退；超过则人工审核 |
| **权益回收** | 退款成立后，Entitlement 置 `revoked`，用户降级 FREE；已产生的练习数据保留 |
| **争议处理** | 保留订单与权益变更全链路日志，便于客诉举证 |

> ⚠️ 永久会员退款需谨慎：商家有义务回收权益，但「已大量使用」可酌情拒绝，需在用户协议中写明。

## 7. 防滥用与合规

| 风险 | 对策 |
|------|------|
| **重复领取试用** | 试用与设备指纹 + 手机号双重绑定，一设备/一号仅一次 |
| **支付伪造/越权开通** | 所有权益开通**必须后端校验渠道订单真实性**，绝不信任客户端回调 |
| **共享账号** | 单账号同时在线设备数限制（如 3 台）；多设备登录冲突策略 |
| **税务合规** | 直连支付需处理发票/代扣代缴；商店内购由商店处理税务 |
| **隐私合规** | 支付信息最小化采集；遵循《个人信息保护法》；支付页独立隐私说明 |
| **未成年人** | 支付前提示；大额支付二次确认；预留家长监护接口 |

## 9. 分阶段实施

| 阶段 | 商业化动作 |
|------|-----------|
| **Phase 1 MVP** | 权益模型/Feature Gate 架构**预留**（见 TDD）；功能暂全免费，埋点埋好 `paywall_shown`（实际不拦） |
| **Phase 2 V1** | 试用期机制上线（注册赠 7 天）；付费墙 UI 接入（免费层限额生效，如评分每日 3 次） |
| **Phase 3 V2** | **永久会员售卖上线**：安卓商店内购 + 微信/支付宝直连；订单系统、权益开通、恢复购买 |
| **Phase 4 远期** | iOS + IAP；全品类 VIP；单内容付费；订阅模式（如需要） |

> **关键**：Phase 1 就把权益模型与 Feature Gate 架构搭好（成本极低），Phase 3 只需"接支付渠道"，不用重构。
