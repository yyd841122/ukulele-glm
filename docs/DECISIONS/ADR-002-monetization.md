# ADR-002：商业化架构 — Feature Gate 统一权限层 + MVP 预留会员模型

> **状态**：已接受（Accepted）
> **日期**：2026-06-26
> **决策者**：技术方案评审（AI 规划 Agent 起草）
> **关联**：`docs/MONETIZATION.md`、`docs/TDD.md` §4.2 + §6.5
> **背景**：用户明确后期要加入会员商用功能，要求文档预留扩展接口。

---

## 1. 背景（Context）

产品规划在 Phase 3 启用会员商用（永久会员 + 试用期，安卓优先，商店内购 + 微信/支付宝直连）。
但首轮技术文档（TDD v1.0）的数据模型与架构中**完全没有会员/订单/权益相关设计**，只在 PRD 提到"会员中心"页面。

若 Phase 1（MVP）直接按"全免费"实现，把权限判断散落在各业务页面（`if(isMember)`），到 Phase 3 接支付时将面临：
- 数据库回溯加 Entitlement/Order/SKU 等表；
- 每个功能页面逐个改造加付费墙；
- 限额功能（如每日评分 N 次）无处落计数；
- 付费墙交互不统一，体验割裂。

需要在 MVP 启动前确定商业化的**架构预留方案**。

---

## 2. 决策（Decision）

采用 **"统一 Feature Gate 权限层 + MVP 阶段预留会员数据模型"** 方案：

1. **数据模型预留**：MVP 即建立 §4.2 全部商业化域表（Entitlement / Order / SKU / TrialGrant / UsageQuota / PaywallEvent），固化 `Tier`、`OrderStatus`、`PaymentChannel` 枚举。
2. **统一权限控制层**：所有受控功能必须经 `FeatureGate.check(FeatureKey)` 统一接口判定，返回 `Granted / GrantedWithQuota / Locked` 三态，**禁止业务页面硬编码会员判断**。
3. **统一付费墙组件**：全 App 单一 `PaywallSheet`，由 `Locked.reason` 驱动。
4. **MVP 阶段策略**：FeatureGate 实现，但所有 FeatureKey **配置为"免费可用"**（即实际全放行），同时埋点 `paywall_shown`。Phase 3 接支付时，仅需把配置改为按权益矩阵判定 + 接支付渠道，**业务页面零改动**。

会员档位、权益矩阵、支付流程、退款规则等业务细节见 `docs/MONETIZATION.md`。

---

## 3. 理由（Rationale）

| 维度 | 说明 |
|------|------|
| **成本极低** | MVP 多建 6 张表 + 一个权限中间件，几乎不增加工作量；收益是 Phase 3 免重构 |
| **关注点分离** | 权限规则集中在 FeatureGate + 权益矩阵配置，业务页面只关心"能不能用"，不关心"会员档位/支付" |
| **配置化降本** | 免费层限额（如评分每日 3 次）、哪些功能收费，全部配置化，运营可调无需发版（理想态） |
| **就高原则清晰** | 用户多权益并存（试用+永久）时，单一接口返回生效档位，客户端逻辑简单 |
| **体验统一** | 付费墙单一组件，转化文案/CTA 一致，利于 A/B 测试与转化率优化 |
| **防掉单/合规前置** | 订单状态机、服务端强校验、防重领试用等关键能力在架构层锚定 |

---

## 4. 后果（Consequences）

### 优点
- Phase 1→3 平滑过渡，接支付零业务代码改动；
- 权益/限额/付费墙规则集中可控、可配置；
- 数据模型一步到位，无后期数据库迁移风险；
- 转化埋点（paywall_shown/dismissed）从 MVP 即采集，提前积累转化数据。

### 缺点 / 代价
- MVP 多写一层 FeatureGate 抽象与 6 张空表，**小幅增加初期复杂度**（可接受）；
- 需要团队遵守"禁止硬编码会员判断"约定（靠 Code Review 保证）；
- 权益矩阵配置需维护（但换来灵活性，值得）。

### 风险缓解
- MVP 阶段 FeatureGate 默认全放行，**不影响任何用户体验**，纯架构预留；
- 提供 FeatureGate 单元测试，保证"免费/试用/会员/限额"四种判定正确；
- 文档（MONETIZATION + TDD §6.5）明确接入契约，降低遗忘风险。

---

## 5. 备选方案（Alternatives Considered）

| 方案 | 为何否决 |
|------|---------|
| **A. MVP 完全不管，Phase 3 再加** | 需回溯改库 + 改所有页面 + 散落判断，重构成本高、易漏 |
| **B. MVP 就接入完整支付** | 支付渠道申请/审核周期长，拖慢 MVP 上线；且无用户验证就收费风险高 |
| **C. 用第三方会员/支付 SaaS** | 抽成高、定制性差、数据不在自己手里；本项目权益模型简单，自建更合适 |

---

## 6. 复核触发条件（Revisit When）

- 接入支付时发现所选渠道（如某商店 IAP）订单模型与 §4.2 Order 字段不兼容 → 调整 Order 结构；
- 业务扩展到订阅制（非永久会员）→ Entitlement.expires_at 与续费逻辑需增强；
- 多乐器全品类 VIP 上线 → Tier 枚举与"按品类授权"逻辑需扩展。
